# Development Notes

## Текущее состояние проекта

- Chrome-расширение подключается к открытой вкладке Яндекс Музыки.
- Локальный сервер связывает расширение и телефон через QR pairing.
- Телефонный пульт умеет:
  - play / pause;
  - предыдущий и следующий трек;
  - лайк и дизлайк;
  - перемотку;
  - управление громкостью;
  - показ следующих треков;
  - выбор вариантов `Моей волны`;
  - запуск `Моей волны по треку`;
  - сброс к обычной `Моей волне`.
- Добавлено автоматическое переподключение телефона после возврата во вкладку, восстановления страницы и появления сети.
- Добавлены trace-логи для расследования редкого сбоя с возможным двойным воспроизведением.

## Архитектура

```text
phone remote
    │
    ▼
HTTP/WebSocket server
    │
    ▼
Chrome extension
    │
    ▼
player bridge + adapted YmExternalAPI
    │
    ▼
internal state of Yandex Music page
```

Главная идея проекта — по возможности использовать внутренние объекты плеера, а не DOM-клики. Это обычно устойчивее к косметическим изменениям страницы.

## Важные файлы

```text
extension/content.js
extension/player-bridge.js
extension/externalAPI/externalAPI.js
extension/externalAPI/extracted-data.js
server/ws_server.js
server_ubuntu/ws_server.js
```

`server/ws_server.js` и `server_ubuntu/ws_server.js` сейчас во многом дублируют друг друга. Любую функциональную правку серверной части важно переносить в обе копии, иначе Windows- и Ubuntu-ветки начнут расходиться.

## Ключевые технические решения

### Встроенный API

- Используется адаптированная копия `YmExternalAPI` в `extension/externalAPI`.
- `replace-webpack.js` получает внутренние объекты страницы, включая `controller` и `playbackController`.
- Варианты `Моей волны` читаются динамически из `rootStore.wheel.items`.

### Выбор варианта Моей волны

Для выбора пресета используется:

```js
Controller.contextController.factory.createContext(...)
Controller.playContext(...)
```

Для `diversity:reshuffle` seed создаётся как:

```text
diversity:reshuffle_<timestamp>
```

### Моя волна по треку

`Моя волна по треку` реализована не через DOM, а через внутренний слой API.

Логика:

1. берётся текущий `trackId`;
2. формируется seed `track:<id>`;
3. создаётся новый vibe-контекст;
4. вызывается `Controller.playContext(...)`.

Это проверено как минимум в двух сценариях:

- текущий трек играет из коллекции;
- текущий трек играет уже внутри `Моей волны`.

### Сброс к обычной Моей волне

Сброс реализован через seed:

```text
user:onyourwave
```

Нативный крестик Яндекс Музыки дополнительно отправляет `queue` уже сыгранных треков. Это, по всей видимости, помогает мягче продолжить рекомендации после сброса, но не требуется для самого переключения режима. Поэтому сейчас поддержка `queue` намеренно не добавлена, чтобы не усложнять рабочую логику без подтверждённой пользы.

Когда стоит вернуться к этому вопросу:

- если после сброса чаще повторяются уже звучавшие треки;
- если рекомендации после сброса ощущаются заметно грубее, чем после нативного крестика;
- если понадобится максимально близко повторить поведение официального интерфейса.

### Навигация

Кнопка `next` защищена: если следующий трек ещё не готов, команда не отправляется. Это нужно, чтобы не ломать внутреннюю очередь `Моей волны`.

Для навигационных команд есть короткий cooldown, чтобы двойной тап не создавал гонки.

### Громкость

Громкость на телефоне меняется только при перетаскивании самого бегунка. Обычный тап по полоске не меняет значение, чтобы случайное касание не делало звук резко тише или громче.

### Переподключение телефона

Телефонный пульт вызывает попытку восстановления связи при:

- `visibilitychange` → вкладка снова стала видимой;
- `pageshow`;
- `online`.

После переподключения пульт отдельно запрашивает актуальное состояние плеера.

## Trace-логи и поиск редкого двойного воспроизведения

### Зачем это добавлено

Однажды был замечен редкий глюк: после команды с телефона появился второй наложенный аудиопоток. Воспроизвести его стабильно пока не удалось, поэтому сейчас важнее не гадать, а поймать точную последовательность событий.

### Что уже логируется

На стороне телефона, сервера и расширения сохраняются:

- клик по кнопке на телефоне;
- получение команды сервером;
- получение команды расширением;
- список доступных вкладок перед командой;
- какая вкладка выбрана для управления;
- состояние плеера до команды;
- состояние сразу после команды;
- дополнительные снимки через `80` и `300` мс;
- обычные события плеера;
- изменение количества реально играющих `<audio>`-элементов;
- событие `audio.multiple-playing`, если найдено больше одного активного аудио.

### Какие вопросы помогут закрыть trace-логи

Если глюк повторится, по логам должно стать видно:

1. была ли команда реально отправлена с телефона;
2. в каком состоянии плеер был перед ней;
3. не выбралась ли неожиданно другая вкладка;
4. не произошло ли рассогласование между видимым `isPlaying` и фактическим поведением аудио;
5. появился ли второй `<audio>` после нашей команды или без неё;
6. не было ли рядом переподключения, смены вкладки или другого внешнего события.

### Ограничение диагностики

`audioPlayingCount` полезен, но не идеален. Если значение равно `0`, это ещё не доказывает, что звука нет. Для разбора причины важна вся временная цепочка, а не одно поле.

## Известные особенности

- Варианты `Моей волны` появляются только после того, как сама страница Яндекс Музыки загрузила `wheel.items`.
- Если на телефоне написано `Варианты пока не загружены`, нужно убедиться, что на ПК открыт экран с вариантами `Моей волны`.
- `prev` внутри `Моей волны` пока работает только в пределах укороченного окна последних треков. На сайте Яндекс Музыки история длиннее.
- `server/ws_server.js` и `server_ubuntu/ws_server.js` необходимо держать синхронными вручную.
- API неофициальное, поэтому после изменений Яндекс Музыки может потребоваться адаптация слоя `extension/externalAPI`.

## Что уже подтверждено проверками

- `Моя волна по треку` запускается из коллекции.
- `Моя волна по треку` запускается из обычной `Моей волны`.
- После запуска `Моей волны по треку` источник меняется на radio/vibe-контекст с `track:<id>`.
- Сброс возвращает `currentId = user:onyourwave`.
- Кнопка сброса отключена, если уже открыта обычная `Моя волна`.
- Телефонный пульт восстанавливается после возврата в браузер не только вручную, но и через автоматическую попытку reconnect.

## Возможные будущие задачи

1. Поймать редкий глюк с двойным воспроизведением в trace-логах и уже по факту решить, нужен ли фикс логики, а не только диагностика.
2. При необходимости добавить поддержку `queue` при сбросе к обычной `Моей волне`.
3. Сделать `prev` для `Моей волны` ближе к поведению сайта Яндекс Музыки.
4. Свести `server` и `server_ubuntu` к одному источнику правды, чтобы не дублировать правки вручную.
5. При необходимости добавить более явную индикацию текущего режима волны на телефоне.

## Как продолжать работу в новом чате

1. Сначала прочитать:
   - `README.md`;
   - `DEVELOPMENT.md`.
2. Затем проверить актуальное состояние репозитория.
3. Если вернёмся к редкому двойному воспроизведению, сначала смотреть `trace-*.jsonl`, а не менять логику вслепую.
4. Если вернёмся к сбросу `Моей волны`, сначала сравнить поведение с нативным крестиком и только потом решать, нужен ли `queue`.
